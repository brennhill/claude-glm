# AIWrap Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the GLM-only wrapper with a generalized `aiwrap` launcher that supports `claude` and `codex` tools against `glm` and `claude` providers, while exposing `glm` as a shell alias over `aiwrap claude glm`.

**Architecture:** Build a neutral provider configuration model under `~/.aiwrap/providers/`, compose provider profiles with tool adapters, and launch the selected local CLI with tool-specific environment derived from provider-neutral config. Keep `aiwrap` as the single real executable and install `glm()` as a shell convenience function.

**Tech Stack:** POSIX shell, bash/zsh shell startup files, optional `jq`, shell-based tests with `expect`

---

## Execution Notes

- Treat `aiwrap` as the only production executable. Do not preserve `bin/glm` as a real wrapper binary.
- Keep generic logic generic. Tool-specific behavior belongs under `scripts/lib/tools/`; provider-specific behavior belongs under `scripts/lib/providers/`.
- Preserve current-working-directory execution behavior for every launch path. `aiwrap` should never change directories.
- Fail explicitly for unsupported tool/provider pairs. Do not infer compatibility from naming.
- Keep setup idempotent and safe to rerun. Token rotation must remain explicit via `--reset-token`.
- Move only the minimum local state needed from the old GLM-specific layout into `~/.aiwrap/providers/glm.json`.
- Install `glm()` as a shell function over `aiwrap claude glm "$@"`, not as an alias string and not as a standalone binary.
